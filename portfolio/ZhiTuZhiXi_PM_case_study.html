1. 项目背景：为什么做这个产品

这个项目不是从“我要做一个看板”开始的，而是从一次真实的求职场景开始的。

我主导过一次就业动员大会，在收集同学反馈时发现，很多同学并不是完全不努力，而是不知道如何把“岗位要求、自己会什么、还缺什么、该补哪门课”连起来。大家会同时看招聘JD、课程平台、学长学姐经验贴和简历模板，但这些信息是割裂的：

  • JD 写的是“熟悉SQL、指标体系、用户增长、A/B测试”，学生不知道这些词对应哪些课程和项目。
  • 课程平台写的是“Python数据分析”“商业分析实战”，学生不知道课程是否真的服务于目标岗位。
  • 老师给我的论文方向是基于 NER 从文本中抽取实体，用于课程与岗位的精准定位，但论文方案本身还停留在算法/研究视角。
  • 就业辅导场景里，老师和学生都需要一个更直观的产品化入口，而不是只看模型结果或论文表格。

因此，我把问题重新定义为：

准备实习或转方向的学生，需要一个能把岗位JD、个人技能、课程学习和可展示项目连接起来的工具，帮助他们从“我该投什么”推进到“我下一步该补什么、做什么、怎么证明”。

这个定义也参考了用户需求陈述的基本写法：明确用户、需求和使用场景，而不是一开始就陷入功能清单。1

2. 用户与场景

目标用户

主用户：

  • 准备投递互联网、数据分析、商业分析、产品、运营、金融科技等实习的学生。
  • 有课程、竞赛或科研经历，但不知道如何对齐岗位JD的人。
  • 需要快速形成求职学习路径和项目包装方向的人。

次级用户：

  • 就业指导老师：希望基于学生目标岗位给出更具体的课程/项目建议。
  • 课程建设或教研人员：希望判断课程内容和市场岗位需求是否匹配。

核心使用场景

  1. 学生看到一个目标岗位，例如“产品经理”“数据产品经理”“商业分析师”。
  2. 学生输入或勾选自己已有技能，例如 Python、Excel、SQL、PPT。
  3. 系统输出岗位画像、技能缺口、课程推荐、相似岗位、GitHub参考项目和可能被面试追问的问题。
  4. 学生据此决定：补什么技能、复现什么项目、简历怎么写、面试怎么讲。

3. 问题定义：我真正要优化什么

原始问题

最初的论文方向聚焦 NER：从岗位文本和课程文本中抽取技能、课程、岗位等实体，并建立映射关系。

如果只按技术问题来做，产品可能会停留在：

  • 抽取准确率是多少？
  • 模型用 BiLSTM-CRF 还是 BERT？
  • 知识图谱怎么建？

但从产品角度看，用户真正关心的是：

  • 我现在适合投什么岗位？
  • 我离目标岗位差哪些能力？
  • 哪些课程或项目能补齐这些能力？
  • 我如何把这个过程变成简历和面试里能讲清楚的证据？

重新定义后的产品问题

如何把 NER 和岗位知识图谱能力，包装成一个低门槛、可解释、能直接指导求职行动的职业路径推荐产品？

评估标准

第一版不追求“全量真实招聘平台级别”，而是优先验证产品闭环：

维度 目标
可理解 学生能看懂岗位画像和缺口技能，不需要理解模型细节。
可行动 输出课程、项目和面试问题，而不是只给匹配分。
可解释 告诉用户为什么推荐这些技能、课程和项目。
可演示 即使没有完整后端和模型，也能在线稳定展示完整流程。
可扩展 后续能接入 NER、Neo4j、薪资预测和真实JD抓取。

4. 产品定位

产品名称

职途智析

产品定位

面向学生和就业指导场景的岗位技能图谱推荐工具。

差异化

普通求职工具常见的输出是岗位列表或简历建议。职途智析的差异点是把“岗位 - 技能 - 课程 - 项目 - 面试问题”放在同一条决策链路里：

  1. 先看目标岗位需要什么。
  2. 再看用户已有技能和岗位要求的差距。
  3. 再把差距映射到课程和开源项目。
  4. 最后把学习路径转成面试可讲的问题和项目证据。

薪资区间 建立岗位预期 demo薪资分布

GitHub参考项目 帮助做作品集/项目复现 按岗位和技能推荐开源项目

模式提示 避免夸大能力 显示 demo / neo4j 模式

第一版暂不做

暂不做 原因
登录和用户账号 不影响验证岗位-技能-课程闭环。
全量实时招聘平台抓取 数据成本和维护成本高，MVP先用可控样例数据。
简历上传解析 涉及隐私和解析准确率，后续再接入。
完整NER模型线上推理 Streamlit公网部署需要轻量稳定，先保留接口和后端替换空间。
精细视觉设计 作品集重点是产品逻辑、流程和可解释结果，不是视觉展示。

这个取舍遵循了作品集案例的核心原则：展示问题、决策、过程和结果，而不是只堆最终界面。2

为什么设计统一接口

我把前端页面和后端模型解耦，封装成统一的 analyze_career(target_job, user_skills) 接口。

这样做有三个好处：

  1. 前端不用关心后端到底是 demo 数据、Neo4j 图谱还是未来的 NER 模型。
  2. 第一版可以快速上线验证用户流程，后续再替换模型。
  3. 面试时可以清楚讲出产品如何从 MVP 过渡到真实数据产品。

7. 核心功能设计

7.1 岗位画像

用户选择目标岗位后，系统展示岗位所属行业、职业路径、常见产出和核心能力。

产品目的：

  • 让用户先理解岗位，而不是直接看一堆技能标签。
  • 把“岗位要求”转成更易理解的工作内容和产出。

7.2 技能差距分析

系统将岗位所需技能和用户已有技能进行对比，输出缺口技能及权重。

示例：

目标岗位：数据分析师
当前技能：Python、Excel
系统识别：缺少 SQL、统计分析、Tableau、A/B测试、业务理解、数据可视化

产品目的：

  • 用户不再只知道“我不够匹配”，而是知道具体差在哪。
  • 技能权重让用户知道先补什么，而不是平均用力。

7.3 课程推荐

课程推荐不是泛泛地推“Python课程”，而是基于缺口技能匹配课程。

产品目的：

  • 将“岗位要求”转化为“学习路径”。
  • 将论文里的课程-岗位精准定位思路落到用户行动上。

7.4 相似岗位推荐

系统根据目标岗位技能集合和其他岗位技能集合的重叠度，推荐可横向投递的岗位。

产品目的：

  • 帮助用户拓宽投递面。
  • 解决学生常见问题：只盯一个岗位，但不知道相近岗位有哪些。

7.5 GitHub参考项目推荐

这是我在产品化时加入的扩展点。学生不仅需要知道“学什么”，还需要知道“做什么项目能证明我学会了”。

因此我把 GitHub 开源项目接入推荐逻辑，用于：

  • 对标成熟数据产品/分析工具。
  • 给学生项目复现和作品集方向。
  • 帮助用户准备面试追问。

7.6 面试问题生成

系统会结合岗位要求和缺口技能生成面试准备问题，例如：

  • 如何定义一个新业务的北极星指标？
  • 如果 JD 要求 A/B 测试，你会如何用自己的项目证明这个能力？
  • 如何把技能差距转化为一个可复盘的学习项目？

产品目的：

  • 把推荐结果从“看完就结束”推进到“可以面试表达”。
  • 让用户形成作品集和简历里的项目讲法。

页面设计取舍

  • 用左输入、右结果的布局，减少用户来回跳转。
  • 第一屏先给岗位画像和关键指标，避免用户直接被大表格淹没。
  • 课程、项目、相似岗位和面试准备放在 Tab 中，保持信息密度但不打断主线。
  • 明确显示“扩展演示数据模式”，把产品能力边界讲清楚。

10. 数据与模型方案

当前 MVP

第一版使用扩展演示数据，保证公网部署稳定：

  • 38 个岗位样例
  • 83 个技能标签
  • 23 个 GitHub 开源参考项目
  • 课程、岗位、技能和项目之间的样例映射

推荐逻辑

输出 计算思路
技能缺口 目标岗位技能集合 - 用户已有技能集合，并保留技能权重。
推荐课程 根据缺口技能匹配课程覆盖范围。
相似岗位 按岗位技能集合重叠度和用户已有技能加权排序。
参考项目 根据目标岗位、项目适配岗位和技能重叠度排序。
面试问题 基于岗位预设问题和高权重缺口技能生成。

后续真实模型接入

论文方向中的 NER 可以作为真实后端的关键模块：

招聘JD / 课程介绍
  ↓
文本清洗
  ↓
NER抽取：岗位、技能、课程、证书、工具、行业
  ↓
实体标准化与去重
  ↓
岗位-技能-课程知识图谱
  ↓
技能缺口与课程推荐

后续可接入：

  • NER技能抽取模型：从 JD 和课程文本中识别技能实体。
  • Neo4j知识图谱：连接岗位、技能、课程和项目。
  • XGBoost薪资预测：结合岗位、城市、技能和经验预测薪资。
  • SHAP解释：解释哪些技能对薪资或岗位匹配影响更大。

NER用于岗位技能抽取是一个成熟研究方向，公开研究也常用 NER 或序列标注方法从招聘文本中识别技能实体。3

11. 最终效果

当前可展示结果

以“数据分析师 + Python/Excel”为例：

  • 系统识别 6 个缺口技能。
  • 给出 13,000 元/月薪资中位数。
  • 返回 8 个相似岗位。
  • 推荐 10 个 GitHub 参考项目。
  • 输出课程建议和面试准备问题。

产品化成果

  • 将原本偏论文/算法的 NER 课程-岗位定位方向，转化为用户可操作的职业路径推荐产品。
  • 搭建了可在线访问的 Streamlit MVP。
  • 通过 adapter 设计预留 demo 与真实图谱后端的切换能力。
  • 把技能差距、课程推荐、项目推荐和面试准备串成闭环。
  • 保留数据产品的可解释边界，没有把 demo 夸大成完整生产系统。

13. 我的关键决策

决策1：先做可运行MVP，而不是先追求完整模型

原因：

  • 求职产品首先要验证用户路径是否成立。
  • 完整 NER + Neo4j + 模型部署成本高，容易陷入技术堆叠。
  • 可运行 demo 更适合作品集和面试展示。

决策2：保留 demo/neo4j 模式提示

原因：

  • 产品作品集不能夸大能力。
  • 面试官会追问真实数据、模型和部署边界。
  • 清楚展示边界反而能体现产品判断和工程常识。

决策3：加入 GitHub 项目推荐

原因：

  • 学生的真实痛点不只是“学什么”，还包括“怎么证明我会”。
  • 开源项目能帮助用户形成作品集和面试谈资。
  • 这让产品从课程推荐延伸到求职成果转化。

决策4：让输出服务于面试表达

原因：

  • 产品经理/数据岗位面试经常追问“你为什么这么做、怎么验证、结果如何”。
  • 所以结果页不只展示分析结果，还要给出面试问题和学习路径。

15. 面试讲法

30秒版本

职途智析是我基于就业动员大会反馈和老师给的 NER 论文方向做的职业路径推荐产品。它解决的是学生看不懂 JD、课程和自身技能之间关系的问题。用户输入目标岗位和已有技能后，系统会输出技能缺口、课程推荐、相似岗位、GitHub项目和面试准备问题。我在第一版里优先验证完整产品闭环，用 demo 数据保证公网可演示，同时通过统一接口预留 NER、Neo4j 和薪资预测模型接入。

2分钟版本

我最初做这个项目，是因为在就业动员大会后收集同学反馈时发现，很多同学不是没有学习资源，而是不知道岗位JD里的技能词和课程、项目之间怎么对应。老师同时给了我一个 NER 相关论文方向，希望用命名实体识别从文本中提取课程和岗位实体，实现更精准的课程-岗位定位。

我没有直接把它做成一个算法展示页，而是先重新定义用户问题：学生需要知道目标岗位要求什么、自己缺什么、该补什么课程、做什么项目能证明能力。基于这个问题，我设计了“目标岗位输入 - 当前技能输入 - 技能缺口 - 课程推荐 - 相似岗位 - GitHub项目 - 面试问题”的产品闭环。

第一版我选择 Streamlit 做 MVP，因为它能快速上线并保证可演示。我把后端封装成 analyze_career 统一接口，并设计 demo 和 Neo4j 两种适配器。这样即使真实图谱或 NER 模型还没上线，前端产品体验也可以先验证；后续接入真实模型时，不需要重写页面。

这个项目对我来说最重要的不是页面，而是把一个论文/算法方向转成了真实可用的产品：它能帮助学生把岗位要求转成学习路径和面试证据，也能为后续就业指导和课程建设提供数据化入口。

16. 面试可能被追问的问题

问题 回答要点
为什么不直接做完整 NER？ 第一版先验证用户流程和推荐闭环，避免模型开发拖慢产品验证；同时保留模型接口。
你的用户洞察来自哪里？ 就业动员大会后的同学反馈、老师论文方向、自己求职准备中对 JD/课程/项目割裂的观察。
这个项目和普通课程推荐有什么区别？ 它不是从课程出发，而是从目标岗位和技能缺口出发，再推荐课程和项目。
如何判断推荐结果好不好？ 短期看分析完成率、课程/项目点击率和反馈；长期看投递准备效率、面试问题准备度和学习路径完成率。
为什么加入 GitHub 项目？ 因为学生最终需要在简历和面试里证明能力，开源项目能帮助形成可展示作品。
如果接入真实数据，最大风险是什么？ 实体抽取准确率、技能同义词归一、课程质量参差不齐、用户隐私和推荐解释性。

17. 作品集摘要

这个案例想展示的不是“我做了一个页面”，而是：

  • 我能从真实场景和用户反馈中定义问题。
  • 我能把论文/算法方向转译成产品方案。
  • 我能做 MVP 取舍，并讲清楚为什么先做哪些、不做哪些。
  • 我能设计产品架构，让 demo 能演示，未来又能扩展。
  • 我能把最终结果转化为用户可执行的学习、项目和面试路径。
